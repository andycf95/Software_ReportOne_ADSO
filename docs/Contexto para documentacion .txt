Estoy desarrollando un software web de gestión de órdenes de mantenimiento correctivo para maquinaria y activos industriales.

Tecnologías:
- Backend: Django
- Frontend: HTML + Bootstrap 5 +CSS
- Base de datos: SQLite en desarrollo
- Patrón: MVT de Django
- Control de versiones: Git, trabajando principalmente en una rama Desarrollo, pasando cambios estables a master se usa rama dependiente de desarrollo rama pytest.
- JavaScript
- AJAX
El módulo principal es Solicitudes de mantenimiento. La entidad principal es Solicitud, que representa una orden de mantenimiento.

Arquitectura actual 
El sistema está desarrollado bajo la arquitectura MVT (Model - View - Template), patrón nativo del framework Django, que separa la lógica de datos, la lógica de negocio y la presentación en tres capas independientes.

Model: define la estructura de la base de datos y las reglas de negocio mediante los modelos Solicitud, Seguimiento, Activo, Sistema, Componente y Usuario.
View: contiene la lógica de negocio y el manejo de peticiones HTTP, procesando los datos y enviándolos al template correspondiente.
Template: capa de presentación construida con HTML, Bootstrap 5 y JavaScript, responsable de renderizar la interfaz de usuario.

El proyecto está organizado en cuatro aplicaciones Django independientes: solicitudes, activos, usuarios y home, cada una con su propio modelo, vistas, URLs y templates. Adicionalmente implementa un patrón de Soft Delete en el modelo Solicitud para conservar la trazabilidad de las órdenes eliminadas, un Custom Manager para filtrar automáticamente los registros eliminados, y una capa de interactividad en el frontend mediante AJAX para la carga dinámica de activos, sistemas y componentes sin recargar la página.

Modelo Seguimiento:
- solicitud: ForeignKey a Solicitud con related_name='seguimientos'.
- comentario: texto.
- fecha: DateTimeField auto_now_add.

El historial de seguimiento se guarda con formato:
"Acción|Descripción"

Ejemplos:
- "Creación de orden|La orden fue creada en estado Pendiente."
- "Cambio de estado|La orden pasó de Pendiente a En proceso."
- "Actualización del proceso|Se realizó inspección inicial."
- "Cambio de estado|La orden pasó de En proceso a Cerrada."

En las vistas, los seguimientos se preparan separando el comentario con partition("|") para mostrar:
- hora a la izquierda,
- línea vertical al centro,
- acción como título,
- descripción debajo en texto más pequeño,
- agrupados por fecha.

- solicitudes_cerradas: muestra solicitudes cerradas, permite filtros por fecha de creación y fecha de cierre.
- detalle_solicitud: muestra información completa e historial.
- crear_solicitud.
- editar_solicitud.
- cambiar_estado_en_proceso: debe ser POST.
- cerrar_solicitud: muestra formulario de cierre y guarda trabajo realizado/observaciones.
- detalle_cierre.

App Usuarios
Se tienen 3 roles en los usuarios técnico, supervisor y administrador

Autenticación y usuarios:
- Se creó una app usuarios.
- Se usa un modelo personalizado Usuario basado en AbstractUser.
- settings.py tiene:
  AUTH_USER_MODEL = 'usuarios.Usuario'
  LOGIN_URL = 'usuarios:login'
  LOGIN_REDIRECT_URL = 'home:index'
  LOGOUT_REDIRECT_URL = 'usuarios:login'
- El usuario tiene campos adicionales:
  numero_identificacion,
  telefono,
  rol.
- AbstractUser ya trae:
  username,
  first_name,
  last_name,
  email,
  password,
  is_staff,
  is_superuser,
  groups,
  user_permissions.
- email se puede sobrescribir como unique=True.
- La Solicitud se relaciona con el usuario creador mediante:
  usuario = ForeignKey(settings.AUTH_USER_MODEL, on_delete=PROTECT, related_name='solicitudes_creadas', null=True, blank=True)
- Se recomienda usar on_delete=PROTECT para no perder trazabilidad.
- Las vistas sensibles deben usar login_required:
  lista_solicitudes,
  solicitudes_cerradas,
  detalle_solicitud,
  crear_solicitud,
  editar_solicitud,
  cambiar_estado_en_proceso,
  cerrar_solicitud,
  detalle_cierre,
  eliminar_solicitud.
- Login y logout pueden manejarse con LoginView y LogoutView de Django.
- Registro tiene vista propia.
- Por seguridad, después del registro es mejor redirigir al login en vez de loguear automáticamente.

Al seleccionar un sistema:
- Se cargan los subsistemas.

Sin necesidad de recargar la página.

Reglas de negocio:
- Una solicitud nace en estado PENDIENTE.
- El flujo válido es PENDIENTE -> EN_PROCESO -> CERRADA.
- No se permite pasar directamente de PENDIENTE a CERRADA.
- Solo las solicitudes PENDIENTES pueden editarse o eliminarse.
- Al pasar a EN_PROCESO, los datos principales quedan bloqueados y se habilitan comentarios de seguimiento.
- Al pasar a CERRADA, la orden queda solo lectura y conserva historial.
- Los cambios de estado se hacen mediante botones/acciones POST, no desde el formulario.
- Las reglas también deben estar protegidas en backend, no solo en frontend.
- El método iniciar_proceso() del modelo valida que solo PENDIENTE pase a EN_PROCESO.
- El método cerrar(trabajo_realizado, observaciones_cierre) valida que solo EN_PROCESO pase a CERRADA, registra fecha de cierre y datos técnicos.
- El método delete() del modelo impide eliminar solicitudes que no estén PENDIENTES.
- No se puede realizar eliminación permanente soft delete (solo pendientes)
- Los  comentarios tampoco se puedes eliminar
- Los activos, sistemas y componentes no se pueden eliminar si tienen solicitudes asignadas
- Los activos solo pueden quedar fuera de servicio y que los sistemas y componentes herede este estado

Filtros:
- Búsqueda por texto usando Q:
  codigo, titulo, descripcion, activo, sistema_activo, subsistema_activo.

- Filtro por criticidad usando los valores enteros de los choices.

- En cerradas también filtro por fecha_cierre.
- Los filtros se aplican antes del paginador.
